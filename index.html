① Data Warehouse & Data Mart — Conceptos y Construcción
🏛 Data Warehouse (DW)
Repositorio central integrado de datos históricos para análisis y toma de decisiones.
01
Definir requerimientos¿Qué preguntas de negocio debe responder? ¿Qué KPIs? Entrevistas con usuarios.
02
Identificar fuentes de datos (OLTP)ERP, CRM, archivos planos, APIs externas. Analizar calidad y estructura.
03
Diseñar modelo dimensionalElegir esquema: Estrella (Star) o Copo de Nieve (Snowflake). Definir hechos y dimensiones.
04
Proceso ETLExtracción → fuentes OLTP. Transformación → limpieza, integración, normalización. Load → cargar al DW.
05
Poblar y validarCarga inicial histórica. Validar integridad con conteos y checksums.
06
Crear capas semánticas / Data MartsSubconjuntos por área (Ventas, RRHH). Cubos OLAP para análisis multidimensional.
07
Publicar y mantenerConectar herramienta BI (Power BI, Pentaho). Programar cargas incrementales.
OLAP Hechos + Dimensiones Esquema Estrella ETL Datos Históricos
🧩 Data Mart
Subconjunto temático del DW orientado a un área de negocio específica.
01
Definir el área de negocioEj: Ventas, Finanzas, Marketing. Delimitar el alcance del Data Mart.
02
Elegir enfoque: Dependiente vs IndependienteDependiente: se alimenta del DW central. Independiente: tiene su propia fuente ETL.
03
Modelar hechos y dimensiones del áreaTabla de hechos (métricas cuantificables) + tablas de dimensiones (contexto: tiempo, producto, cliente).
04
ETL específico del Data MartExtraer desde DW (si dependiente) o fuentes OLTP. Transformar al modelo dimensional definido.
05
Construir cubos OLAPAgregaciones pre-calculadas para drill-down, roll-up, slice & dice en reportes.
06
Conectar a herramienta BIPower BI, Pentaho BI, Tableau. Definir roles y permisos por área.
DW → general, toda la empresa. Data Mart → específico, un departamento.
② Metodología Hefesto — Paso a Paso Completo
🔥 Metodología Hefesto (DW Iterativo)
Metodología latinoamericana para construcción de DW. Orientada a requerimientos del usuario. 4 grandes etapas.
ETAPA 1 · ANÁLISIS
1.1
Identificar preguntas del negocioEntrevistas con usuarios. ¿Qué necesitan analizar?
1.2
Identificar indicadores y perspectivasIndicadores: métricas (ventas, cantidad). Perspectivas: dimensiones (tiempo, producto, región).
1.3
Modelo conceptualDiagrama de estrella conceptual: indicadores en centro + perspectivas alrededor.
ETAPA 2 · ANÁLISIS OLTP
2.1
Explorar OLTP origenMapear tablas de la BD transaccional. Buscar donde residen indicadores y perspectivas.
2.2
Conformar indicadoresVerificar si se calculan directamente o requieren transformación (suma, promedio, etc.).
2.3
Nivel de granularidad¿Al detalle de qué se almacena? Ej: por día, por factura, por sucursal.
ETAPA 3 · MODELO LÓGICO
3.1
Tabla de hechosColumnas = indicadores (métricas FK a dimensiones). Granularidad definida.
3.2
Tablas de dimensionesAtributos descriptivos de cada perspectiva. Surrogate key (SK) artificial.
3.3
Completar diagrama estrella/snowflakeVerificar integridad referencial y cardinalidad entre hechos y dimensiones.
ETAPA 4 · INTEGRACIÓN
4.1
Diseñar ETLMapeo OLTP → DW. Transformaciones, limpieza de nulos, homogenización.
4.2
Carga inicialPoblar dimensiones primero (SCD si aplica), luego tabla de hechos.
4.3
Carga incrementalProgramar actualizaciones periódicas (diarias, semanales). Detectar cambios en fuente.
Hefesto es iterativo: al terminar etapa 4 se vuelve a etapa 1 para nuevos requerimientos. Preguntas del negocio → modelo → ETL → BI.
③ Metodología DDD — Domain-Driven Design para NoSQL
🌿 Metodología DDD (Domain-Driven Design)
Diseño orientado al dominio del negocio. Fundamental para modelar datos en NoSQL según cómo se accede, no cómo se almacena relacionalmente.
CONCEPTOS CLAVE DDD
▸
DominioEl área de negocio que se modela (ej: "Ventas Online").
▸
Entidad (Entity)Objeto con identidad única y ciclo de vida. Ej: Cliente, Pedido.
▸
Objeto de Valor (Value Object)Sin identidad propia, define por sus atributos. Ej: Dirección, Precio.
▸
Agregado (Aggregate)Conjunto de entidades y value objects con raíz. La raíz garantiza invariantes. Ej: Pedido (raíz) + Ítems.
▸
RepositorioAbstracción de acceso a datos. Cada agregado tiene un repositorio.
▸
Bounded ContextLímite explícito donde un modelo domina. Ej: Contexto "Envíos" vs "Facturación".
▸
Ubiquitous LanguageVocabulario común entre devs y negocio dentro del bounded context.
PASOS PARA APLICAR DDD EN NoSQL
01
Entender el dominioEntrevistas con expertos del negocio. Construir el Ubiquitous Language.
02
Definir Bounded ContextsSeparar dominios con límites claros. Cada contexto puede tener su propia BD.
03
Identificar Agregados y sus raícesDefinir qué entidades van juntas. La raíz del agregado = unidad de consistencia.
04
Modelar según patrones de accesoEn NoSQL: diseñar el modelo según las QUERIES, no la normalización. Desnormalizar si es necesario.
05
Elegir BD NoSQL según el agregadoDocumentos complejos → MongoDB. Grafos de relaciones → Neo4j. Clave-valor simple → Redis. Series temporales/alta escala → Cassandra.
06
Definir repositorios por agregadoInterfaces de acceso: findById, save, delete. El repositorio oculta la BD subyacente.
07
Context Mapping (si hay múltiples contextos)Definir cómo se comunican: Shared Kernel, Anti-Corruption Layer, Published Language.
En NoSQL: el Agregado define el documento. Guarda todo lo que se consulta junto, EN EL MISMO DOCUMENTO.
④ Arquitectura BigData — Capas y Estrategias
🌊 Arquitectura BigData (Lambda / Kappa)
Diseño de sistemas para procesar grandes volúmenes, variedad y velocidad de datos (las 5 V's).
LAS 5 V's DEL BIGDATA
V1
VolumenTerabytes, Petabytes de datos. Requiere almacenamiento distribuido (HDFS, S3).
V2
VelocidadDatos en tiempo real o near-real-time. Kafka, Spark Streaming.
V3
VariedadEstructurados, semi-estructurados (JSON, XML) y no estructurados (imágenes, texto).
V4
VeracidadCalidad e incertidumbre de los datos. Necesario validar y limpiar.
V5
ValorEl fin último: extraer conocimiento accionable del dato.
ARQUITECTURA LAMBDA (3 CAPAS)
L1
Batch Layer (capa fría)Procesa TODOS los datos históricos. Alta latencia, alta precisión. Ej: Hadoop MapReduce, Spark Batch.
L2
Speed Layer (capa caliente)Procesa datos en tiempo real. Baja latencia, datos recientes. Ej: Spark Streaming, Flink, Storm.
L3
Serving Layer (capa de servicio)Combina resultados de Batch + Speed. Responde queries. Ej: HBase, Cassandra, ElasticSearch.
Lambda: batch para exactitud histórica + speed para frescura reciente. Complejidad: mantener 2 pipelines.
ARQUITECTURA KAPPA (SIMPLIFICADA)
K1
Un solo pipeline de streamingTodo pasa por la capa de velocidad. No hay capa batch separada.
K2
Log inmutable como fuente de verdadKafka como log de eventos. Permite re-procesar desde el inicio.
K3
Serving LayerResultados del stream se vuelcan a BD de consulta. Más simple de operar.
PASOS PARA PLANTEAR UNA ARQUITECTURA BD
01
IngestaDefinir fuentes y conectores (Kafka, Flume, Sqoop, APIs).
02
Almacenamiento crudo (Data Lake)HDFS, S3, Azure Blob. Guardar datos sin transformar.
03
ProcesamientoBatch (Spark) o Streaming (Flink). Limpiar, transformar, enriquecer.
04
Almacenamiento analíticoData Warehouse (Hive, BigQuery) o NoSQL según tipo de query.
05
Consumo y visualizaciónBI tools, APIs REST, ML pipelines, dashboards.
⑤ Criterios de Selección de BD NoSQL — Resumen por Tipo
🔵 Cuándo usar cada tipo de BD NoSQL
Criterios de decisión basados en el modelo de datos, patrón de acceso y requisitos del sistema.
TIPO MODELO ÚSALO CUANDO... FORTALEZA EVITAR SI...
Documental
MongoDB, CouchDB		 Documentos JSON/BSON anidados Datos semi-estructurados, esquema variable, jerarquías complejas (catálogos, perfiles, CMS) Consultas ricas, índices secundarios, flexibilidad de esquema Relaciones complejas entre entidades (usa grafos) o transacciones multi-documento críticas
Clave-Valor
Redis, DynamoDB		 Hash table distribuida Caché, sesiones, contadores, colas, configuraciones. Acceso por ID exacto, ultra-rápido Latencia sub-milisegundo, escalado horizontal simple Necesitas consultar por campos distintos a la clave o relaciones entre datos
Grafo
Neo4j		 Nodos + Relaciones + Propiedades Redes sociales, recomendaciones, detección de fraude, rutas, cualquier dato altamente relacional Traversal de relaciones profundas sin JOINs costosos Datos no relacionales o consultas simples por clave
Columnar
Cassandra, HBase		 Familias de columnas (filas + columnas dinámicas) Series temporales, IoT, logs, escritura masiva y continua, alta disponibilidad geográfica Escritura extremadamente rápida, escalado lineal, tolerancia a fallos Consultas ad-hoc complejas o esquema muy cambiante por fila
Regla DDD → el tipo de BD NoSQL se elige por el Agregado y su patrón de acceso, NO por el volumen ni la popularidad.
